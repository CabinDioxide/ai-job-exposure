Global AI Occupational Exposure Monitor

AI影响就业全球监督

Overview 概述

This document describes the methodology, data sources, and known limitations of the Global AI Occupational Exposure Monitor. We measure how AI affects jobs across 12 countries using observed Claude usage data from Anthropic's Economic Index.

本文档描述"AI影响就业全球监督"项目的方法论、数据来源及已知局限性。我们使用Anthropic经济指数中的Claude实际使用数据，测量AI对12个国家就业的影响。

Data Sources 数据来源

Three exposure dimensions:

  • Theoretical AI Exposure (0-10): LLM capability assessment from Karpathy's analysis — "Can AI technically do this job?" Shared across all countries.
  • Observed AI Exposure (0-10): Computed from real Claude usage data. For the US, from Anthropic's job_exposure.csv (756 SOC occupations, released 2026-03-05). For non-US countries, from V4 country-level onet_task_pct distributions (released 2026-01-15, data from 2025-11-13 to 2025-11-20).
  • Automation vs Augmentation ratio: From V3 release (2025-08-04 to 2025-08-11), per-country ratio of directive/automated vs collaborative/augmentative AI usage.

Employment data: ILOSTAT where available; otherwise estimated ISCO major group distributions based on ILO World Employment and Social Outlook and country income level.

三个暴露维度：

  • 理论AI暴露度 (0-10)：基于Karpathy的LLM能力评估——"AI技术上能不能影响这个职业？"所有国家共用同一套分数。
  • 实测AI暴露度 (0-10)：基于Claude的真实使用数据。美国数据来自Anthropic的job_exposure.csv（756个SOC职业，2026年3月5日发布）。非美国国家数据来自V4国别onet_task_pct分布（2026年1月15日发布，数据采集期2025年11月13-20日）。
  • 自动化 vs 增强比率：来自V3版本（2025年8月4-11日），各国AI使用中指令式/自动化与协作式/增强式的比率。

就业数据：优先使用ILOSTAT；不可用时按ILO《世界就业和社会展望》及国家收入水平估算ISCO大类分布。

Computation Pipeline 计算管道 

Step 1: Extract each country's onet_task_pct distribution from V4
Step 2: Map task text → SOC codes via onet_task_statements.csv (18,428 tasks)
Step 3: For each SOC: raw_score = Σ(task_pct × task_penetration)
Step 4: Normalize: country_exposure[soc] = (country_raw / us_raw) × us_official_exposure
Step 5: SOC → ISCO-08 crosswalk (839 SOC → 436 ISCO)
Step 6: SOCs with no country data → exposure = 0
步骤1：从V4数据中提取每个国家的onet_task_pct分布
步骤2：通过onet_task_statements.csv将任务文本映射到SOC职业代码（18,428个任务）
步骤3：对每个SOC计算：raw_score = Σ(task_pct × task_penetration)
步骤4：归一化：country_exposure[soc] = (country_raw / us_raw) × us_official_exposure
步骤5：SOC → ISCO-08交叉映射（839个SOC → 436个ISCO）
步骤6：无数据的SOC → 暴露度 = 0

Known Limitations 已知局限性

  1. Claude ≠ All AI. V4 data captures only Claude usage. GitHub Copilot, ChatGPT, Midjourney, and domain-specific AI tools are invisible. Countries where Claude has low market share will appear underexposed.
  2. Claude users ≠ the labor force. Claude users skew toward knowledge workers, English speakers, and technology-adjacent roles. India's 58,098 conversations are overwhelmingly from IT professionals. India's 560 million workers in agriculture, construction, and informal services are invisible in this data. The data tells us about the IT sector's AI exposure, not India's.
  3. Sparse coverage in small markets. Philippines has 89 tasks (6% of US). This means 352 of 436 ISCO occupations receive zero exposure — not because AI doesn't affect them, but because we have no data. Mexico (94 tasks, 6.5%) has the same problem.
  4. Single-platform, single-week snapshot. V4 covers one week (Nov 13-20, 2025) of one platform. Usage patterns may vary by season, product releases, or market events.
  5. SOC taxonomy mismatch. O*NET task statements use SOC 2010; job_exposure.csv uses SOC 2018. Some major 2018 occupations (e.g., 15-1252 Software Developers) have no task mapping. Mitigated at ISCO level but causes SOC-level gaps.
  6. No within-country sector decomposition. A software developer at TCS Bangalore and at a village startup get the same exposure score. Sector-level data exists in V3/V4 but is not yet integrated.
  7. Employment data is estimated. Non-US employment distributions are ISCO major group estimates, not granular occupation-level census data. This affects weighted averages.
  1. Claude ≠ 所有AI。V4数据只捕获Claude的使用。GitHub Copilot、ChatGPT、Midjourney及领域专用AI工具均不可见。Claude市场份额低的国家会显得暴露不足。
  2. Claude用户 ≠ 劳动力。Claude用户偏向知识工作者、英语使用者和技术相关岗位。印度的58,098次对话绝大多数来自IT从业者。印度5.6亿农业、建筑和非正规服务业工人在这份数据中是不可见的。数据告诉我们的是IT行业的AI暴露，而非印度整体的。
  3. 小市场覆盖稀疏。菲律宾只有89个任务（美国的6%）。这意味着436个ISCO职业中有352个暴露度为零——不是因为AI不影响它们，而是因为我们没有数据。墨西哥（94个任务，6.5%）也有同样的问题。
  4. 单平台、单周快照。V4覆盖一个平台（Claude）的一周数据（2025年11月13-20日）。使用模式可能因季节、产品发布或市场事件而变化。
  5. SOC分类体系错位。O*NET任务描述使用SOC 2010；job_exposure.csv使用SOC 2018。一些主要的2018年职业代码（如15-1252软件开发人员）没有任务映射。在ISCO层面有所缓解，但在SOC层面存在缺口。
  6. 无国内行业分解。在班加罗尔TCS工作的软件开发人员和在村镇创业公司的开发人员获得相同的暴露分数。V3/V4中存在行业级数据但尚未整合。
  7. 就业数据为估算值。非美国国家的就业分布基于ISCO大类估算，而非细粒度的职业级人口普查数据。这影响加权平均值。

Why We Stopped Using Adjustment Factors 为什么不再使用调整因子

The earlier version (V1) adjusted US exposure scores using three factors: digital infrastructure (ITU IDI), informality rate (ILO), and AI adoption (IMF + Tortoise). We abandoned this approach because:

  1. The factors were national averages applied uniformly to all occupations. India's 89% informality was applied equally to software engineers and street vendors.
  2. The informality discount values (how much informality reduces exposure per occupation category) were expert judgment, not empirically derived.
  3. With V4 country-level Claude usage data available, adjustment factors became redundant. The V4 data already implicitly captures infrastructure (poor infrastructure → fewer Claude users), informality (informal workers don't use Claude), and AI adoption (low adoption → fewer conversations). Multiplying by external indices on top of this would be double-counting.

早期版本（V1）使用三个因子调整美国暴露分数：数字基础设施（ITU IDI）、非正规就业率（ILO）和AI采用率（IMF + Tortoise）。我们放弃了这种方法，原因如下：

  1. 这些因子是全国平均值，被统一应用到所有职业上。印度89%的非正规就业率被同等地施加到软件工程师和街头小贩身上。
  2. 非正规就业折扣值（非正规就业在多大程度上降低每类职业的暴露度）是专家判断，而非实证推导。
  3. 有了V4国别级Claude使用数据后，调整因子变得多余。V4数据已经隐含了基础设施（基础设施差→Claude用户少）、非正规性（非正规工人不用Claude）和AI采用率（采用率低→对话量少）。在此基础上再乘外部指数属于重复计算。

  • build_country_exposure.py — V4 data → per-country per-SOC/ISCO exposure
  • build_global_data.py — assembles 12-country dataset with employment weights
